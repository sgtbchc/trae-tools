## Context

Trae IDE 是一款 AI 驱动的编程工具，用户在对话编程过程中无法直观感知 token 消耗和 API 额度使用情况，也无法在不离开当前工作流的情况下了解 AI 任务执行状态。当前 Trae IDE 具有扩展 API 体系，但缺少围绕 Trae ID 的插件生态和系统级状态展示能力。

本项目需要构建两个核心系统：
1. **插件系统**：运行在 Trae IDE 内部，提供 token 统计和 API 额度管理
2. **Windows 灵动岛应用**：独立桌面应用，通过 IPC 与 Trae IDE 通信，提供系统级状态展示

## Goals / Non-Goals

**Goals:**
- 构建可扩展的插件架构，支持 token 统计和 API 额度管理作为首批插件
- 实现 Windows 灵动岛应用，提供仿苹果灵动岛的视觉体验和交互能力
- 通过 Trae ID 统一身份认证，确保插件和灵动岛与 Trae 生态的兼容性
- 预留多平台扩展接口，使 API 额度管理可扩展至其他大模型平台
- 灵动岛支持多显示器环境，不影响其他应用正常使用

**Non-Goals:**
- 不实现 macOS/Linux 版本的灵动岛（仅 Windows）
- 不实现插件市场/商店功能
- 不实现 token 计费和付费功能
- 不实现 Trae IDE 核心功能的修改
- 不实现灵动岛的自定义主题/皮肤系统（当前版本）

## Decisions

### 1. 插件架构：基于事件总线的微内核模式

**选择**：采用微内核 + 事件总线架构，插件通过标准接口注册和通信。

**替代方案**：
- VS Code 式 Extension Host 进程隔离：过于重量级，增加 IPC 复杂度
- 简单函数调用式插件：缺乏解耦，难以扩展

**理由**：微内核模式提供良好的扩展性和隔离性，事件总线使插件间松耦合通信成为可能，同时保持轻量级实现。

### 2. 灵动岛技术栈：Tauri + React

**选择**：使用 Tauri 2.0 作为应用框架，React 作为 UI 框架。

**替代方案**：
- Electron：体积大（~150MB），内存占用高
- 原生 WinUI 3：开发效率低，跨平台能力差
- WPF：技术栈老旧，与现代 Web 开发脱节

**理由**：Tauri 体积小（~5MB），性能优秀，Rust 后端提供系统级 API 调用能力，React 前端生态成熟。Tauri 2.0 支持多窗口和系统托盘，适合灵动岛场景。

### 3. Trae IDE 与灵动岛通信：WebSocket + 本地 HTTP

**选择**：Trae IDE 侧启动 WebSocket 服务器，灵动岛作为客户端连接；HTTP 端点用于简单查询和操作。

**替代方案**：
- Named Pipe：Windows 专用，调试不便
- 共享内存：复杂度高，数据同步困难
- gRPC：过于重量级

**理由**：WebSocket 提供双向实时通信，适合任务状态推送；HTTP 端点用于简单操作（如取消任务、审批确认），两者互补。本地通信无需考虑网络延迟。

### 4. 数据存储：本地 JSON 文件

**选择**：使用本地 JSON 文件存储所有数据（token 记录、API Key、额度缓存、配置等），存放在应用数据目录下。

**替代方案**：
- SQLite：功能强大但引入额外依赖，对于本地工具场景过于重量级
- IndexedDB：仅浏览器环境，灵动岛无法共享
- LevelDB：无 SQL 查询能力，二进制格式不可读

**理由**：JSON 文件零依赖、人类可读、易于调试和手动编辑。本地工具场景下并发写入风险极低（单用户单进程），通过文件锁和原子写入（写临时文件后 rename）保证数据完整性。数据目录结构如下：
```
~/.trae-tool/
├── token-records.json      # Token 消耗记录
├── quota-cache.json        # API 额度缓存
├── config.json             # 全局配置（含 API Key）
└── plugin-state.json       # 插件状态
```

### 5. API Key 存储：本地配置文件

**选择**：API Key 直接存储在 `config.json` 中，不做额外加密。

**理由**：本地开发工具场景，用户自行管理本机安全即可，无需引入系统 Keychain 等额外依赖。简化实现，降低复杂度。

### 6. 灵动岛窗口管理：Windows DWM API

**选择**：通过 Tauri 的 Rust 后端调用 Windows DWM（Desktop Window Manager）API 实现窗口置顶、透明、无边框、点击穿透等效果。

**理由**：DWM API 是 Windows 官方窗口合成器接口，可实现灵动岛所需的视觉和交互效果，包括圆角、阴影、动画等。

## Risks / Trade-offs

- **[Trae IDE 扩展 API 限制]** → Trae IDE 可能不暴露完整的 token 统计和任务状态 API，需要调研可用接口，必要时通过拦截通信层获取数据
- **[灵动岛窗口与全屏应用冲突]** → 某些全屏应用可能覆盖灵动岛，需实现 DWM Z-order 管理和全屏检测逻辑
- **[多显示器 DPI 差异]** → 不同显示器可能使用不同缩放比例，需处理 Per-Monitor DPI Aware 渲染
- **[WebSocket 连接稳定性]** → Trae IDE 重启或休眠可能导致连接断开，需实现自动重连和心跳机制
- **[火山 Coding Plan API 变更]** → 第三方 API 可能变更，需设计适配层隔离变化
- **[Tauri 2.0 稳定性]** → Tauri 2.0 相对较新，可能遇到未文档化问题 → 使用稳定发布版，关注社区更新
